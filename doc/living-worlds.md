# Living Worlds

**Last Updated:** 2026-04-28

Autonomous world evolution that makes scenes feel alive between player visits. The world clock advances in real time, the sun moves across the sky, and narrative events are generated by Claude Haiku and broadcast to all connected viewers. A World Journal panel in the viewer records what happened while the player was away.

No new Marble generation. No per-tick cost unless an event fires. All world mutation uses lightweight DB writes and code-gen paths already in the renderer.

---

## Architecture Overview

```
WorldHeartbeat (10-min real tick)
  ├── for each living scene (livingEnabled=true):
  │     ├── newTime = (worldTime + 600) % 86400
  │     ├── sceneManager.updateEnvironment(sceneId, { worldTime: newTime })
  │     ├── bus.publish → world_time_update
  │     └── 25% chance + eventStore present:
  │           ├── eventStore.getRecentEvents(sceneId, 3)  ← avoid repetition
  │           ├── generateWorldEvent(scene, recentEvents) ← Claude Haiku, 8s timeout
  │           ├── eventStore.addEvent(worldEvent)
  │           └── bus.publish → world_event
  │
  └── only ticks when bus.activeSessions().length > 0

Viewer (SplatViewer.tsx)
  ├── world_time_update WS message
  │     └── window.__setWorldTime(t) → applyWorldTimeLighting(t)
  │           └── lerp between keyframes → hemiLight + sunLight intensity/color
  └── world_event WS message
        ├── window.__addWorldEvent(e) → WorldJournal prepends to list
        └── SplatViewer onNewEvent → badge++ (if journal closed)
```

### Tick Math

| Constant | Value | Effect |
|---|---|---|
| `TICK_INTERVAL_MS` | 600 000 ms (10 min) | Real-time tick cadence |
| `GAME_SECONDS_PER_TICK` | 600 s | Game-time advance per tick |
| 1 game-day | ~24 real hours | 144 ticks × 600 game-sec = 86 400 |
| `EVENT_CHANCE` | 0.25 | 25% per scene per tick → ~3.6 events/day per scene |

## Data Model

### EnvironmentConfig additions (`src/scene/types.ts`)

| Field | Type | Default | Description |
|---|---|---|---|
| `worldTime` | `number?` | `undefined` | 0–86 400 seconds since midnight. `undefined` = static scene (no heartbeat) |
| `livingEnabled` | `boolean?` | `false` | Opt-in per scene. Heartbeat skips scenes where this is falsy |

### WorldEvent (`src/storage/types.ts`)

| Field | Type | Description |
|---|---|---|
| `eventId` | `string` | UUID |
| `sceneId` | `string` | Owning scene |
| `worldTime` | `number` | Game time when the event occurred |
| `eventType` | `string` | `weather` / `discovery` / `npc_activity` / `anomaly` |
| `headline` | `string` | ≤60 chars, shown in journal list |
| `body` | `string` | 1–2 sentences of flavor text, shown on expand |
| `createdAt` | `number` | Unix timestamp (ms) |

## Backend Modules

### `src/world/world-heartbeat.ts`

Modeled on `src/npcs/npc-heartbeat.ts`. Spawns a `setInterval` timer, calls `timer.unref()` so the process can exit cleanly if no other work is pending.

```typescript
export function startWorldHeartbeat(
  sceneManager: SceneManager,
  bus: RealtimeBus,
  eventStore?: WorldEventRepository,
): () => void
```

**Key design decisions:**
- Skips tick entirely when `bus.activeSessions().length === 0` — no API cost when no one is online
- `eventStore` is optional; if absent, time still advances but no events are generated
- `Promise.allSettled` at both session and scene level — one failing scene never blocks others
- `updateEnvironment()` does not bump scene version or create a version snapshot; it is a background state mutation invisible to the user's edit history

### `src/world/event-generator.ts`

```typescript
export async function generateWorldEvent(
  scene: Scene,
  recentEvents: WorldEvent[],
): Promise<Omit<WorldEvent, "eventId" | "sceneId" | "createdAt"> | null>
```

- Model: `claude-haiku-4-5-20251001` (lowest cost, ~200-token prompt)
- Prompt injects: scene title, description, NPC names (up to 5), current time-of-day string (localized to Chinese), last 3 event headlines (to avoid repetition)
- Returns `null` on any failure (timeout, parse error, unexpected JSON) — heartbeat silently skips, worldTime update still completes
- Hard timeout: 8 s via `Promise.race`

### `src/scene/scene-manager.ts` — `updateEnvironment()`

```typescript
async updateEnvironment(sceneId: string, patch: Partial<EnvironmentConfig>): Promise<Scene>
```

Shallow-merges `patch` into `scene.sceneData.environment`. Writes directly to the scene repository. Does **not** increment `version`, does **not** write to `scene_versions`. This keeps the version history clean for user-visible edits while still persisting worldTime across server restarts.

### Storage

| File | Backend | Schema |
|---|---|---|
| `src/storage/sqlite/world-event-repo.ts` | SQLite | `world_events` table, index on `(scene_id, created_at DESC)` |
| `src/storage/postgres/world-event-repo.ts` | PostgreSQL | Same schema, postgres tagged templates, `BIGINT` for timestamps |

Both implement `WorldEventRepository`:
```typescript
interface WorldEventRepository {
  init(): Promise<void>;
  addEvent(event: WorldEvent): Promise<void>;
  getRecentEvents(sceneId: string, limit?: number): Promise<WorldEvent[]>;
}
```

### API Endpoint

`GET /scenes/:sceneId/events?limit=20`

Returns `{ events: WorldEvent[] }` ordered by `created_at DESC`. Subject to the same share-token / session auth guard as other scene routes. Used by WorldJournal on open to load history.

## Viewer — Day/Night Lighting

### Why SplatViewer, not scene-renderer.ts

`scene-renderer.ts` calls `executeCode(sceneCode)` for every scene, which zeroes out `hemiLight.intensity` and `sunLight.intensity` as part of the code-gen sandbox setup. Any lighting applied before `executeCode` would be overwritten. World-time lighting is therefore applied at the SplatViewer level using Three.js lights added to the root scene before the renderer mounts.

### Keyframes (`WT_KF` in `SplatViewer.tsx`)

| Time | Sun color | Sun intensity | Hemi intensity | Description |
|---|---|---|---|---|
| 0 / 86 400 s | `#2244aa` | 0.05 | 0.15 | Deep night |
| 21 600 s (06:00) | `#ff8c42` | 0.80 | 0.35 | Dawn |
| 43 200 s (12:00) | `#fff4e0` | 1.20 | 0.70 | Noon |
| 64 800 s (18:00) | `#ff8c42` | 0.80 | 0.35 | Dusk |

`applyWorldTimeLighting(t, hemiLight, sunLight)` linearly interpolates between the two surrounding keyframes. Color channels are interpolated separately (R, G, B extracted from hex).

### Callback pattern

`window.__setWorldTime` follows the existing `window.__moveNpc` / `window.__emoteNpc` pattern for bridging App.tsx's WebSocket event handling into SplatViewer's Three.js scene. The callback is registered after the Three.js lights are created and deleted in the `useEffect` cleanup.

## Viewer — World Journal

### Component: `viewer/src/components/WorldJournal.tsx`

| Prop | Type | Description |
|---|---|---|
| `sceneId` | `string` | Used to fetch event history on open |
| `isOpen` | `boolean` | Controls visibility (panel renders null when closed, but component stays mounted) |
| `onClose` | `() => void` | Called by the × button |
| `onNewEvent` | `(e) => void?` | Called on each pushed event — SplatViewer uses this to increment badge |

**Event flow:**
1. Component mounts → registers `window.__addWorldEvent`
2. App.tsx receives `world_event` WS → calls `window.__addWorldEvent(event)`
3. WorldJournal prepends event to its local list (max 50); calls `onNewEvent(event)`
4. SplatViewer's `onNewEvent` increments `journalBadge` if journal is closed
5. On open: `setJournalBadge(0)` resets badge; `GET /scenes/:id/events` loads full history

### Toggle button

Positioned `bottom: 12, right: 12` (z-index 40), always visible when `status === "ready" && sceneId`. Shows badge overlay (max displayed: "9+") when new events arrived with journal closed. Badge resets on open.

## Enabling Living Worlds on a Scene

Set `livingEnabled: true` and optionally seed `worldTime` via the agent tool or a direct `PATCH /scenes/:id` call:

```json
{
  "sceneData": {
    "environment": {
      "livingEnabled": true,
      "worldTime": 43200
    }
  }
}
```

Or in the create-scene tool:

```typescript
sceneData: {
  environment: {
    livingEnabled: true,
    worldTime: 43200,  // start at noon
  }
}
```

## Areas for Improvement

### 1. Event Quality & Continuity

**Current:** Each event is generated with 3 recent headlines as context. No persistent world narrative thread.

**Improvement:** Maintain a rolling "world state summary" (100–200 tokens) updated after each event. Feed this summary into the next event prompt. This creates story arcs: a storm starts in one event, intensifies in the next, and clears two ticks later. Store the summary in `EnvironmentConfig.worldNarrative` (string field).

### 2. NPC Reactions to World Events

**Current:** World events and NPC heartbeats are independent systems.

**Improvement:** After generating a world event, pass the headline and body into `npc-heartbeat.ts` as `perceptionContext`. NPCs will spontaneously comment on what happened in the world. This reuses the existing `PerceptionBus` / `spontaneousNpcLine` path with minimal code change.

### 3. Variable Tick Rate

**Current:** Fixed 10-minute real-time tick regardless of scene activity.

**Improvement:** Use an exponential backoff: scenes with active sessions tick every 10 minutes; scenes with no recent visitors tick every 60–120 minutes. This cuts API cost for inactive scenes while keeping active scenes responsive. Track `lastViewedAt` on the scene or session.

### 4. Player-Triggered Events

**Current:** World events are purely autonomous (heartbeat-driven).

**Improvement:** Allow player actions to seed events. When a player interacts with a special object or completes a quest-like interaction, trigger an immediate `generateWorldEvent` call with the player's action as additional context. The event is then visible to all other players in that scene. This closes the loop between player agency and world narrative.

### 5. worldTime Persistence Across Restarts

**Current:** `worldTime` is persisted to the scene record on every heartbeat tick. If the server restarts mid-tick, up to 10 minutes of in-game time is lost.

**Improvement:** Store the last real-time wall-clock timestamp alongside `worldTime`. On heartbeat startup, compute how many ticks' worth of time passed during downtime and fast-forward `worldTime` accordingly (capped at, say, 2 game-days to avoid flooding events after a long outage).

### 6. Multi-Scene Time Zones

**Current:** Each scene has an independent `worldTime` clock.

**Improvement:** Support a "universe time" concept where scenes share a common epoch but apply offsets (`timeZoneOffset?: number`). All scenes in the same universe are synchronized: if it's midnight in one city scene, it's noon in a scene marked as the "opposite hemisphere." This is purely additive — heartbeat applies `(universeTime + offset) % 86400` instead of per-scene time.

### 7. Event Archive & Player History

**Current:** `world_events` keeps all events indefinitely; `getRecentEvents` fetches the last 20 by default.

**Improvement:** Add an archival pass (weekly cron) that summarizes old events into condensed "lore entries" — longer-form historical records stored in a separate `world_lore` table. The World Journal could show both recent events and lore, making long-lived scenes feel like they have real history. Old raw events beyond 30 days can be pruned.

### 8. Admin / Debug Controls

**Current:** No way to manually trigger a tick or force-generate an event without changing constants.

**Improvement:** Add a `POST /admin/scenes/:sceneId/tick` endpoint (admin-only, guarded by a server secret) that immediately runs one heartbeat tick for the given scene. Useful for testing the Living Worlds pipeline without waiting 10 minutes.

### 9. Viewer: Smooth worldTime Interpolation

**Current:** worldTime jumps by 600 game-seconds every 10 real minutes. The sun position updates discretely.

**Improvement:** After receiving `world_time_update`, animate `worldTime` smoothly over ~30 seconds using a `requestAnimationFrame` loop in SplatViewer (lerp from current to target). The jump is currently unnoticeable because sun lighting is subtle, but becomes jarring if tick rate is increased or if the viewer starts mid-day.

### 10. Weather as a Renderer Effect

**Current:** Weather events are narrative only (text in the journal).

**Improvement:** Map `eventType: "weather"` to visual changes: rain → particle system via sceneCode injection, fog → scene fog density increase, clear sky → remove effects. The `world_event` WS message already carries `eventType` and `headline`, so the viewer could parse keywords ("暴雨", "大雾") and apply corresponding sceneCode via `executeCode()` in a sandboxed iframe overlay.
